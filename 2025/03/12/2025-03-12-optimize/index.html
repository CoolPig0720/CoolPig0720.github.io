性能测试核心指标解释

发表于2025-03-12|更新于2025-03-12|技术文档
|总字数:5k|阅读时长:16分钟|浏览量:

总述

优化无止境，优化的最高境界是不渲染。
优化的本质就是不渲染或少渲染或用更省的方法渲染。
知识网站：移动游戏性能优化通用技法

1 性能测试需要关注哪些指标

FPS、Jank、BigJank、平均内存、峰值内存、网络收发流量、CPU 使用率等

2 FPS

含义：Frames Per Second，每秒渲染的帧数，表示应用（尤其是游戏或 UI 动画）的画面流畅度。
正常范围：移动设备目标为 60 FPS（高端设备可达 90-120 FPS），稳定在 50-60 FPS 以上用户体验良好。低于 30 FPS 会明显卡顿。
异常原因：主线程阻塞、复杂渲染、GPU/CPU 负载过高、内存泄漏导致 GC 频繁、资源加载慢等。
常见检测方法：
• Android Studio Profiler（CPU/GPU 渲染轨迹）。
• adb 命令：adb shell dumpsys gfxinfo <package> 计算帧时间和 Jank。
• 第三方工具如 PerfDog、GameBench。
• 开发者选项中启用”Profile GPU rendering”显示柱状图。

3 CPU 占用与 GPU 占用

CPU 占用

含义：应用进程占用 CPU 资源的百分比，反映计算强度。
正常范围：闲置时<10%，运行时平均<50-60%，峰值<90%。游戏场景可更高，但需避免长期高占用。
**异常原因**：主线程复杂计算、无效循环、频繁 GC、后台任务过多、代码未优化（如过度线程）。
**常见检测方法**：
• Android Studio Profiler（CPU 轨迹）。
• adb 命令：adb shell top -m 10 或 adb shell dumpsys cpuinfo | grep <package>。
• PerfDog 等工具实时监控多核占用。

GPU 占用

含义：图形处理器占用率，负责渲染图像、动画等。
正常范围：游戏中 50-80% 正常，长期 100% 会导致过热降频。
异常原因：过度 Draw Call、高分辨率纹理、复杂着色器、Overdraw（过度绘制）。
常见检测方法：
• Android Studio Profiler（GPU 轨迹，Android 10+ 支持）。
• 开发者选项”Profile GPU rendering”。
• PerfDog 或专用 GPU 监控工具显示负载和温度。

Draw Calls
参考：什么是 Draw Calls-CSDN 博客
Draw Calls 是指 CPU 向 GPU 发出的绘制命令。每个 Draw Call 告诉 GPU 如何渲染一部分场景，包括使用哪些顶点、纹理、着色器等。简单来说，Draw Call 是 CPU 与 GPU 之间的通信桥梁，用于指示 GPU 绘制特定的图形。优化 Draw Calls 的方法有：批处理、精灵图集 Sprite Atlas 合并图标纹理、启用 GPU Instancing。
SetPass Calls（SetPass 调用） 是渲染管线中核心的性能指标之一，本质是 GPU 切换渲染状态（着色器 Pass、材质、纹理、渲染目标等）的次数，直接反映渲染效率的高低。

对比维度 SetPass Call Draw Call
核心作用 绑定渲染状态（Shader Pass、材质参数） 提交网格绘制指令，渲染单个 Mesh
执行顺序 先执行，必须在 Draw Call 之前完成 后执行，完全依赖 SetPass 设置好的渲染状态
单条指令开销 极高（触发 GPU 管线冲刷，重置渲染状态） 极低（仅 CPU 端命令提交，无额外 GPU 开销）
触发条件 切换 Shader Pass、切换材质/渲染状态 同一渲染状态下，每新增 1 个独立未合批 Mesh
性能优化优先级 最高（渲染 CPU 开销的核心来源） 次高（仅数量爆炸时才会成为瓶颈）
合批的影响 合批无法减少 SetPass Call 数量，反而要求同一 SetPass 下才能合批 合批的核心目标，就是减少 Draw Call 数量

4 内存占用

含义：
• PSS(Proportional Set Size) ：比例共享内存，最准确反映应用实际物理内存占用（共享库按比例分配）。
• Native Heap：C/C++ 代码分配的内存（如 Bitmap、OpenGL 资源）。
• Managed Heap(Java Heap 或 Dalvik/ART Heap) ：Java/Kotlin 对象分配的内存，受 GC 管理。
正常范围：低端设备 PSS<100-200MB，中高端<300-500MB。重复操作后 PSS 不应持续上升>10-20%。
异常原因：
• 内存泄漏（对象未释放）。
• 大量 Bitmap 或资源加载。
• 频繁分配大对象导致 GC。
• Native 代码泄漏。
常见检测方法：
• Android Studio Memory Profiler (查看 Heap Dump、分配追踪，区分 Java/Native)。
• adb 命令：adb shell dumpsys meminfo <package> 显示 PSS、Native Heap、Java Heap 等。
• LeakCanary 库自动检测泄漏。

补充：VSS RSS PSS USS
• VSS：虚拟内存总量，含未实际分配的物理内存，数值最大，无实际参考意义
• RSS：实际驻留物理内存，含全部共享内存，存在重复计算问题
• PSS：按进程数均摊共享内存的物理占用，最贴合真实内存消耗
• USS：进程完全独占的物理内存，退出进程可立即释放
核心关注指标：常规性能测试主看PSS，评估进程真实内存占用和系统内存压力，定位内存泄漏专看USS，其持续上涨可直接判定泄漏；一般来说内存占用大小有如下规律：VSS >= RSS >= PSS >= USS

5 卡顿 (Jank 与 BigJank)

含义:
满足两个条件记作一次 Jank：
• 当前帧耗时 > 两倍的前三帧平均耗时
• 当前帧耗时 > 两倍的电影帧耗时
满足两个条件记作一次 BigJank:
• 当前帧耗时 > 两倍的前三帧平均耗时
• 当前帧耗时 > 三倍的电影帧耗时

电影帧耗时是指每一帧画面渲染或显示所需的时间，通常以毫秒为单位计算，标准电影帧率为 24 帧每秒（FPS），每帧的耗时约为 41.67 毫秒。

正常范围：Jank 率<2-5%，帧时间稳定<16ms。
**异常原因**：主线程阻塞（IO、网络、复杂布局）、GC 暂停、过度动画、帧时间不均（Stutter 指帧时间波动大）。
**常见检测方法**：
• Android Studio Profiler（Frames 轨迹，>16ms 帧标红）。
• Systrace 或 Perfetto 追踪帧生命周期。
• adb shell dumpsys gfxinfo 计算 Jank 次数。
• PerfDog 显示 Jank 次数和大卡顿（Big Jank）。

6 功耗/发热

含义：应用导致的电池消耗和设备温度上升，高功耗常伴随发热。
正常范围：闲置<10mW，运行时视场景（游戏可达数 W），15 分钟场景平均功耗不应过高导致温度>45°C。
异常原因：频繁唤醒（WakeLock）、网络请求、GPU/CPU 高负载、位置服务、亮屏时间长。
异常表现：快速掉电、设备烫手、系统降频导致二次卡顿。
常见检测方法：
• Android Studio Energy/Power Profiler（Pixel 6+ 支持 Power Rails 细分）。
• Battery Historian 分析历史耗电事件。
• adb dumpsys battery 或第三方如 PerfDog 估算瞬时功耗。
• 温度：adb shell cat /sys/class/thermal/thermal_zone*/temp。

7 流量

含义：应用消耗的移动数据或WiFi流量（上传 + 下载）。
正常范围：视功能，前台使用合理（如视频高流量正常），后台应最小化（<几 MB/小时）。
异常原因：频繁请求、未压缩数据、广告加载、后台同步过多、泄漏请求。
常见检测方法：
• Android Studio Network Profiler（实时流量统计）。
• adb 命令：TrafficStats API 或计算两次 uid 流量差。
• 系统设置中查看应用流量，或第三方工具抓包（Wireshark+tcpdump）。

8 TBDR 与 IMR 架构

8.1 TBDR（Tile-Based Deferred Rendering，基于瓦片的延迟渲染）架构

详细解释
TBDR 是一种专为移动和嵌入式设备优化的 GPU 渲染架构，主要由 Imagination Technologies（如 PowerVR 系列）开发，并在苹果的 A 系列和 M 系列芯片中广泛采用。它将渲染过程分为两个主要阶段，以最大化效率并最小化内存带宽和功耗消耗：

  • 几何处理阶段（Geometry Processing / Tiling）：
    ◦ GPU 接收所有几何数据（顶点、索引等），但不立即渲染。
    ◦ 将整个帧缓冲区（framebuffer）分成小瓦片（tiles，通常 16x16 或 32x32 像素）。
    ◦ 对所有几何体进行排序和 binning（分组），确定每个瓦片中包含哪些图元（primitives，如三角形）。
    ◦ 这个阶段只处理顶点着色器（vertex shader），生成一个”参数缓冲区”（parameter buffer），存储每个瓦片的图元列表。
    ◦ 隐藏表面移除（Hidden Surface Removal, HSR）在这里初步进行，剔除完全不可见的图元。
  • 片段处理阶段（Fragment Processing / Rendering）：
    ◦ 对于每个瓦片，GPU 使用片上内存（on-chip memory）加载相关图元。
    ◦ 执行片段着色器（fragment shader），但只对可见像素进行着色（deferred shading）。
    ◦ 由于瓦片小巧，所有操作都在高速片上内存中完成，避免频繁访问外部 DRAM。
    ◦ 最终将渲染结果写入帧缓冲区。
    TBDR 的核心优势在于”延迟”（deferred）：它在着色前进行精确的可见性测试，避免对被遮挡像素的无效计算。这特别适合移动设备，因为它显著降低过绘制（overdraw，即多次渲染同一像素）和内存带宽需求。举例：在复杂场景中，IMR 可能渲染 4-5 倍的像素深度，而 TBDR 只渲染可见层。

适用场景与特点
• 优点：低功耗、低带宽需求、高效处理不透明几何体。支持高级功能如无损压缩，进一步节省资源。
• 缺点：几何阶段的 binning 可能增加 CPU 开销；不适合高度透明或动态场景（需多次渲染通行）。
• 示例实现：PowerVR GPU、Apple 的 AGX GPU（M1/M2 芯片）。在软件层面，支持 OpenGL ES、Vulkan 和 Metal API。

8.2 IMR（Immediate Mode Rendering，即时模式渲染）架构

详细解释
IMR 是传统桌面和高端 GPU 的主流架构（如 NVIDIA GeForce、AMD Radeon），强调实时性和灵活性。它采用流水线式处理，几何体一经提交即立即渲染，没有明显的阶段分离：

  • 几何处理：
    ◦ 顶点数据直接传入 GPU，执行顶点着色器生成图元。
    ◦ 图元立即进入光栅化（rasterization），转换为片段（fragments）。
  • 片段处理：
    ◦ 每个片段立即执行片段着色器、深度测试（Z-buffer）和混合（blending）。
    ◦ 结果直接写入帧缓冲区，可能多次访问外部内存（读/写深度缓冲区、颜色缓冲区）。
    ◦ 支持早 Z 测试（early Z-test）来部分减少无效着色，但不如 TBDR 彻底。
    IMR 的”即时”特性意味着渲染管线是连续的：CPU 提交命令，GPU 立即处理。这使得它适合复杂、动态场景，但会导致高内存带宽消耗，因为每个像素可能被多次覆盖（overdraw）。现代 IMR GPU 通过缓存（如 ROP 缓存）和压缩算法缓解这个问题，但仍不如 TBDR 高效在带宽受限的环境中。

适用场景与特点
• 优点：简单、直观，易于调试；支持复杂着色器和多渲染通行；高性能在高带宽桌面环境中。
• 缺点：高功耗、高带宽需求；过度绘制问题在复杂场景中突出，导致浪费计算资源。
• 示例实现：NVIDIA、AMD、Intel 的桌面 GPU；部分移动 GPU 如 Qualcomm Adreno（虽有 TBR 变体，但核心接近 IMR）。

8.3 TBDR 与 IMR 的区别

以下是两个架构的核心区别，使用表格便于比较：

方面 TBDR (Tile-Based Deferred Rendering) IMR (Immediate Mode Rendering)
渲染流程 分两个阶段：先 tiling（几何排序到瓦片），后 deferred rendering（延迟着色，只渲染可见像素）。 单一连续流水线：几何立即渲染，片段即时处理和写入帧缓冲区。
内存带宽 极低，使用片上内存处理瓦片；参数缓冲区存储图元列表，避免多次外部访问。 较高，频繁读/写外部帧缓冲区；依赖缓存和压缩缓解，但仍易受 overdraw 影响
过绘制处理 优秀，通过 HSR 在着色前完全移除隐藏表面；只着色可见像素。 一般，早 Z 测试可减少，但仍可能多次渲染被遮挡像素。
功耗与效率 低功耗，适合移动设备；节省带宽 50-70% 在复杂场景中。 高功耗，适合桌面/高性能场景；效率依赖硬件带宽。
适用平台 移动/嵌入式（如手机、平板、Apple Silicon）；强调效率。 桌面/游戏机（如 PC、主机）；强调性能和灵活性。
复杂性 硬件更复杂（需 tiling 硬件），软件优化需考虑瓦片边界。 硬件简单，软件开发更直观；易于多线程。
性能瓶颈 几何阶段的 binning 开销大；在透明物体多时效率下降。 Overdraw 和带宽瓶颈；在高分辨率下易过热。
示例厂商 Imagination (PowerVR)、Apple (AGX)。 NVIDIA、AMD、Intel；部分 Qualcomm (Adreno)。

总结区别的核心点
• 效率 vs. 灵活性：TBDR 牺牲一些即时性换取极致效率（延迟渲染避免浪费），适合带宽/功耗受限的移动场景；IMR 优先即时处理，适合高性能需求，但效率较低。
• 历史演变：IMR 是早期标准，TBDR 是移动领域的创新（源于 1990s 的 PowerVR），如今苹果等公司将其扩展到桌面（如 M 系列芯片）。
• 开发影响：在 TBDR 上，开发者需优化几何提交顺序和减少透明渲染；在 IMR 上，焦点在减少 overdraw（如排序图元）。

9 常见性能瓶颈的解释与分析

9.1 GPU 侧瓶颈

1. DrawCall（绘制调用）

含义：CPU 向 GPU 发出的渲染命令，每次调用指示 GPU 绘制一组几何体。
为什么是瓶颈：过多导致 CPU 频繁准备状态、GPU 上下文切换开销大。
影响：GPU/CPU 占用升高、FPS 下降、功耗增加。
常见场景：大量独特材质对象、动态批处理失败。
分析方法：Unity Profiler/Unreal Insights查看DrawCall/SetPass数量；RenderDoc捕获帧。
优化建议：批处理（Static/Dynamic Batching）、纹理图集（Atlas）、GPU Instancing、MaterialPropertyBlock。

静态合批与动态合批

  1. 核心原理差异
  • 静态合批：在项目构建阶段，对标记为 Static 的静态物体的网格进行合并，生成一个合并后的大网格与材质包，运行时仅提交 1 次 DrawCall，全程不再修改，CPU 开销极低。
  • 动态合批：在运行时实时对符合条件的动态物体进行网格合并，每帧都要执行合批计算，CPU 有额外开销，无需提前标记 Static。
  1. 适用场景与限制
  • 静态合批：适用于场景中位置、旋转、缩放全程不变的物体（如场景地形、建筑、固定装饰），无顶点数上限限制，但会增加包体与内存占用；禁止运行时修改物体 Transform，否则合批失效。
  • 动态合批：适用于场景中频繁移动 / 变换的小型物体（如特效、UI 元素、动态道具），仅支持顶点数低于 900 的网格，且材质必须完全一致；合批由引擎自动完成，无需额外操作。
  1. 核心选型原则：固定不动的物体优先用静态合批降 DrawCall；频繁变动的小物体用动态合批，避免过度合批导致 CPU 开销反超 GPU 收益。

2. OverDraw（过度绘制）

含义：同一像素被多次渲染（如半透明叠加）。
为什么是瓶颈：浪费 GPU 填充率，移动端 Fill Rate 有限。
影响：GPU 占用激增、发热严重。
常见场景：UI 层叠、粒子系统、重叠几何体。
分析方法：开发者选项”Show overdraw”；Frame Debugger 显示红色区域。
优化建议：前后排序渲染、减少半透明、扁平化 UI 布局。

3. Shader 复杂度（着色器复杂度）

含义：着色器指令数、分支、纹理采样过多。
为什么是瓶颈：增加每个像素/顶点的计算周期。
影响：GPU 占用高、帧时间延长。
常见场景：PBR 光照、多光源、复杂后处理。
分析方法：RenderDoc 分析 Shader 周期；查看指令计数。
优化建议：简化 Shader、LOD 切换、预计算光照、减少分支。

4. 粒子特效（Particle Effects）

含义：大量粒子模拟烟雾、火、雨等效果。
为什么是瓶颈：高 DrawCall + OverDraw + CPU 位置更新。
影响：CPU/GPU 双高、帧率波动。
常见场景：爆炸、天气、技能特效。
分析方法：Particle System Profiler 查看粒子数与渲染时间。
优化建议：限制粒子数、GPU 粒子（Compute Shader）、合并渲染、序列帧替代。

5. Fill Rate（填充率）瓶颈

含义：GPU 每秒填充像素能力受限。
为什么是瓶颈：高分辨率 + 高 OverDraw + 多采样耗尽。
影响：GPU 100% 占用、FPS 锁定。
常见场景：高分辨率屏幕、后处理（Bloom、AA）。
分析方法：降低分辨率观察改善；GPU Profiler 查看像素处理时间。
优化建议：动态分辨率、减少后处理、压缩纹理格式。

6. 带宽瓶颈（Memory Bandwidth）

含义：访问外部内存速率限制。
为什么是瓶颈：移动端带宽低，频繁纹理采样/帧缓冲读写受限。
影响：GPU 等待内存、功耗高。
常见场景：4K 纹理、多渲染目标（RT）。
分析方法：Profiler 内存拷贝时间；降低纹理分辨率观察。
优化建议：ASTC/ETC2 压缩、降低 RT 分辨率。

7. 顶点数量过多（Vertex Overload）

含义：每帧处理的顶点数过多。
为什么是瓶颈：顶点着色器与皮肤计算开销大。
影响：CPU/GPU 双高。
常见场景：高精度模型、复杂动画角色。
分析方法：Profiler 查看 Vertices/Tris 计数（目标<200k-500k/帧）。
优化建议：LOD 系统、模型简化、GPU Skinning。

9.2 CPU 侧瓶颈

1. UI 重建（UI Rebuild）

含义：UI 布局变化触发网格重建。
为什么是瓶颈：主线程频繁重计算顶点/UV。
影响：主线程卡顿、Jank 增多。
常见场景：动态文本更新、嵌套 Canvas。
分析方法：Profiler UI 模块查看 Rebuild/Batch 时间。
优化建议：扁平化 Canvas、缓存静态 UI、最小化变化。

2. GC 频繁触发（Garbage Collection）

含义：托管内存临时对象过多导致回收暂停。
为什么是瓶颈：暂停主线程数 ms 至数十 ms。
影响：规律性卡顿、帧时间 spike。
常见场景：字符串拼接、LINQ、未使用对象池。
分析方法：Profiler GC Alloc 与时间；帧时间图规律 spike。
优化建议：对象池、避免装箱、使用 struct、预分配数组。

3. 主线程阻塞（IO/网络/同步操作）

含义：主线程执行阻塞任务。
为什么是瓶颈：直接阻塞渲染与输入。
影响：严重卡顿（>50ms）。
常见场景：同步加载资源、网络请求。
分析方法：Profiler 主线程长任务；Systrace 追踪。
优化建议：全部异步（Coroutine/async-await、线程）。

4. 物理计算过重

含义：物理引擎模拟计算量大。
为什么是瓶颈：FixedUpdate 固定时间步占用 CPU。
影响：CPU 高、帧率不稳。
常见场景：大量刚体、布料模拟。
分析方法：Profiler Physics Processing 时间。
优化建议：简单碰撞体、分层碰撞、调整 Fixed Timestep、睡眠阈值。

5. 音频开销

含义：大量音效解码与混合。
为什么是瓶颈：音频线程或主线程 CPU 占用。
影响：CPU 上升、场景切换卡顿。
常见场景：密集枪战音效、未压缩音频。
分析方法：Profiler 音频模块。
优化建议：压缩格式（OGG）、限制同时声音数、预加载。

9.3 内存/IO 侧瓶颈

1. 资源加载（Asset Loading）

含义：纹理、模型等资源从磁盘/网络加载到内存/GPU。
为什么是瓶颈：IO 阻塞、内存峰值、GC 触发。
影响：启动/关卡切换卡顿、内存溢出。
常见场景：高清纹理、首次加载大场景。
分析方法：Profiler 加载时间与内存分配。
优化建议：异步加载（Addressables）、压缩、分包、流式加载。
