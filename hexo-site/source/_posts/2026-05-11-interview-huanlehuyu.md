## 什么是静态合批？什么是动态合批？什么是实例化？

### 概念定义

- **静态合批**：在项目构建阶段，对标记为 `Static` 的静态物体的网格进行合并，生成一个合并后的大网格与材质包。运行时仅提交 **1 次 DrawCall**，全程不再修改，**CPU 开销极低**。

- **动态合批**：在运行时实时对符合条件的动态物体进行网格合并，每帧都要执行合批计算，**CPU 有额外开销**，无需提前标记 Static。

- **实例化（GPU Instancing）**：同一个 Mesh、同一个 Material，只上传一次模型数据，GPU 重复绘制多份。适用于草、树、子弹、重复小怪、同类特效等。

### 适用场景与限制对比

| 特性 | 静态合批 | 动态合批 | 实例化 |
|------|---------|---------|--------|
| **执行时机** | 构建阶段 | 运行时每帧 | 运行时 |
| **CPU 开销** | 极低 | 较高 | 低 |
| **顶点限制** | 无上限 | 低于 900 | 无硬性限制 |
| **材质要求** | 可不同 | 必须完全一致 | 必须完全一致 |
| **Transform 修改** | 禁止（会失效） | 允许 | 允许 |
| **适用对象** | 地形、建筑、固定装饰 | 特效、UI、小型动态物件 | 大量重复物体 |
| **额外代价** | 包体与内存占用增加 | CPU 每帧计算 | 需要相同 Mesh + Material |

> **注意**：动态合批由引擎自动完成，开发者无需额外操作，但需确保物体满足条件（顶点数、材质一致）。

## 多人团战，比如说 50 人团战，哪些东西可以做动态合批？

> **核心原则**：材质相同、顶点少、频繁出现的小物体可以做动态合批。

具体可分为以下几类：

- **UI 类**：血条、小图标、飘字、伤害数字等
- **小型动态物件**：掉落物、小道具、简单投射物
- **小特效**：小火花、小碎片、小范围粒子、简单光效等

## 如何测一个产品的 LOD 分级？

> **LOD 核心思想**：根据物体与相机的距离动态切换不同精度的模型——近处使用高精度模型，远处使用低精度模型，在保证视觉效果的同时显著提升渲染性能。

测试 LOD 的核心是看**不同机型、不同距离、不同画质**下，资源复杂度如何变化。以王者技能为例，测试思路如下：

### 测试步骤

1. **确定测试维度**
   - 模型：面数、Object 数量
   - 特效：粒子数量、粒子生命周期、粒子发射率
   - 渲染：Draw Call、Overdraw、Shader 复杂度
   - 真机性能：帧率、内存、CPU/GPU 时间

2. **固定测试条件**
   - 同机型、同分辨率、同场景，确保变量唯一

3. **测试不同 LOD 档位**
   - **LOD-高**：全粒子、全 Shader、完整模型
   - **LOD-中**：减少粒子、降低贴图精度
   - **LOD-低**：简化特效、关闭后处理

4. **分析 Overdraw**
   - 工具：RenderDoc、Frame Debugger
   - 场景：单特效 → 组合特效 → 多人团战实战
   - 重点观察：火焰、烟雾、半透明光效的大量透明堆叠

5. **分析粒子和 Object**
   - 关注指标：粒子峰值、粒子同屏数、Object 数量、Draw Call 变化

## 如何使用 RenderDoc 分析竞品渲染策略？

以获取王者荣耀孙悟空的面数、粒子数量、贴图大小为例：

1. **截帧**：固定视角下抓取孙悟空释放技能的一帧
2. **定位**：通过 Event Browser 定位角色、技能特效以及相关 Draw Call
3. **分析角色模型面数**
   - 进入 **Mesh Viewer**
   - 查看 Vertices（顶点数）、Indices（索引数）、Triangles（三角面数）
4. **分析粒子系统与特效压力**
   - 重点查看 Transparent Pass、特效相关 Draw Call 数量
   - 关注 Quad 数量以及 Overdraw 情况
5. **分析贴图大小与压缩策略**
   - 进入 **Texture Viewer**
   - 查看贴图分辨率、压缩格式（ASTC、ETC2 等）、Mipmap、显存占用
6. **分析 Shader 与渲染流程**
   - 进入 **Pipeline State**
   - 查看 Shader Pass 数量、Blend 模式、深度测试、后处理流程、Render Target 切换
7. **综合分析**
   - 结合 Draw Call、模型复杂度、粒子数量、Overdraw、Shader 复杂度及贴图规格，评估整体渲染优化策略和 LOD 资源控制方案

## 如何用 Unity Profiler 分析函数耗时问题？

> **案例**：函数 `MinigameTestUpdate` 耗时 68ms，60FPS 对应 16.6ms，已掉到十几帧，属于**严重卡顿**。

### 分析步骤

1. **查看调用树指标**
   - **Self Time 高** → 函数自身逻辑过重
   - **Total Time 高** → 子函数调用链存在问题
   - 关注 Calls（调用次数）、GC Alloc（GC 分配）

2. **展开调用链**
   - 逐层深入，找到真正的耗时热点
   - 重点排查：高频 GC、高频 Instantiate/Destroy、频繁调用、大量循环

3. **TimeLine 分析**
   - 判断是**主线程卡**、**渲染线程卡**，还是其他原因

4. **辅助工具验证**
   - 必要时结合 Deep Profiler、Frame Debugger 等工具进一步定位

## 如何使用 Memory Profiler 分析内存？

### 分析维度

1. **拍摄内存快照**
   - 使用 Memory Profiler 抓取目标场景的快照

2. **Texture（纹理）**
   - 关注：Size（内存占用）、Format（压缩格式）、Resolution（分辨率）、Mipmap
   - 目的：判断纹理是否合理、是否做了压缩

3. **Mesh（网格）**
   - 关注：Vertex Count、Index Count、Mesh Size
   - 目的：判断面数是否过高、是否存在重复 Mesh

4. **RenderTexture（RT）**
   - 重点排查：是否泄漏、分辨率是否过高、是否长期驻留

5. **Reference（资源引用链）**
   - 分析：为什么资源没释放、谁引用了它、是否存在内存泄漏

6. **Mono / Lua 内存**
   - Mono Heap（C# 托管内存）
   - Lua Heap（Lua 脚本内存）
   - Native（引擎原生内存）

## 什么是超频，什么是降频？

| 特性 | 超频 | 降频 |
|------|------|------|
| **定义** | 让 CPU/GPU 运行频率超过默认频率 | 系统主动降低 CPU/GPU 频率 |
| **触发原因** | 用户/系统主动开启高性能模式 | 温度过高、功耗过高 |
| **目的** | 提升性能 | 保护硬件、降低温度 |
| **典型场景** | 高帧率模式、狂暴模式、电竞模式 | 长时间游戏后手机发热 |
| **代价/结果** | 发热增加、功耗增加、更容易降频 | FPS 下降、卡顿、掉帧、抖动帧 |
